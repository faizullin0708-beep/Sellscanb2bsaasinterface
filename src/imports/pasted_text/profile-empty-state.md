Ты — senior product designer, UX/UI designer и SaaS product architect.

Спроектируй production-ready интерфейс SellScan без предзаполненных демо-данных.

Проект:
SellScan — SaaS-сервис для продавцов на Wildberries, Ozon и Яндекс Маркете.

Цель продукта:
Сервис помогает продавцам маркетплейсов видеть реальную операционную прибыль по SKU, находить товары с отрицательной маржой, контролировать остатки, возвраты, комиссии, логистику и получать ежедневные отчеты.

Главная задача этого дизайн-задания:
Переделать интерфейс так, чтобы после регистрации новый пользователь НЕ видел никаких фиктивных, демо или заранее заполненных данных.

Сейчас в интерфейсе ошибочно отображаются демо-данные:
- Иван Петров
- ИП
- ivan@company.ru
- +7 (999) 123-45-67
- ООО "Компания"
- Москва
- ИНН 7701234567
- КПП 770101001
- ОГРН 1234567890123
- юридический адрес “Москва, ул. Примерная, д. 1, 123456”
- Growth Plan
- 45 дней в системе
- 3 активных подключения
- 289 товаров под контролем
- 5 активных алертов
- демо-активность вроде “Обновлена себестоимость для 15 SKU”, “Подключен маркетплейс Ozon”, “Экспортирован отчет по товарам”, “Изменен тариф на Growth”

Этого быть не должно.

Нужно спроектировать интерфейс нового пользователя, где:
- показываются только реальные данные;
- если данных нет, показывается empty state;
- пользователь сам заполняет профиль;
- пользователь сам добавляет реквизиты компании;
- пользователь сам подключает маркетплейсы;
- пользователь сам импортирует себестоимость;
- статистика считается только из реальной базы;
- история активности появляется только после реальных действий;
- тариф показывается только реальный, а не захардкоженный.

Главный UX-принцип:
Если данных нет — показываем понятное пустое состояние и действие.
Если данные есть — показываем реальные данные.
Никогда не показываем демо-данные в production-интерфейсе.

Общий стиль:
- темный SaaS-интерфейс;
- современный B2B dashboard;
- аккуратные карточки;
- много воздуха;
- понятные empty states;
- без визуального мусора;
- без фейковых значений;
- интерфейс должен выглядеть как настоящий production-продукт, а не демо-макет;
- дизайн должен быть готов к передаче frontend-разработчику.

Цветовая логика:
- фон приложения: почти черный / dark navy;
- карточки: темно-серые;
- primary accent: фиолетово-синий, как в текущем SellScan;
- success: зеленый;
- warning: желтый/оранжевый;
- danger: красный;
- muted text: серый;
- borders: тонкие темные границы;
- empty state icons: приглушенные, не кричащие.

Навигация:
Слева sidebar с пунктами:
- Личный кабинет
- Дашборд
- Товары

13. ПОВЕДЕНИЕ КНОПОК

Кнопка “Заполнить профиль”:

Кнопка “Добавить реквизиты компании”:

- открывает EditCompanyModal с текущими реальными значениями.

Кнопка “Подключить маркетплейс”:
- ведет на `/app/connections`;
- или открывает modal выбора маркетплейса.

Кнопка “Импортировать CSV”:
- ведет на `/app/costs`;
- или открывает modal импорта.

Кнопка “Настроить Telegram”:
- ведет на `/app/settings/telegram`.

Кнопка “Выбрать тариф”:
- ведет на `/app/billing`.

Кнопка “Повторить” в error state:
- повторяет загрузку данных.

--------------------------------------------------
14. ТРЕБОВАНИЯ К RESPONSIVE
--------------------------------------------------

Desktop:
- sidebar слева;
- контент в 2 колонки:
  - основная колонка: onboarding, profile, company;
  - правая колонка: stats, activity, quick actions.

Tablet:
- sidebar может быть collapsed;
- карточки перестраиваются в 1 колонку или 2 узкие.

Mobile:
- sidebar превращается в bottom navigation или drawer;
- все карточки идут одной колонкой;
- формы открываются fullscreen modal;
- кнопки на всю ширину.

--------------------------------------------------
15. ТРЕБОВАНИЯ К ACCESSIBILITY
--------------------------------------------------

- Контраст текста должен быть достаточным для dark theme.
- Inputs должны иметь labels.
- Error messages должны быть рядом с полями.
- Buttons должны иметь visible focus state.
- Disabled actions должны объясняться через tooltip или helper text.
- Empty states не должны быть только иконками — нужен текст.
- Цвет не должен быть единственным индикатором статуса.
